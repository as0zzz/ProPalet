# Поэтапный план разработки проекта: offline-first мобильное приложение для грузчиков

## 1. Назначение документа

Этот документ разбивает разработку проекта на **много маленьких последовательных шагов**, чтобы нейросеть или разработчик могли идти по ним поэтапно, а не пытаться сгенерировать и собрать весь проект за один проход.

Документ нужен как **план реализации + чек-лист самопроверки**.

Главная идея:
- выполнять проект маленькими законченными итерациями;
- после каждого шага обязательно проверять результат;
- не переходить к следующему шагу, пока текущий шаг не стабилен;
- постоянно смотреть на ошибки компиляции, линтера, типизации и нативной сборки;
- не оставлять «потом доделаем» в фундаментальных частях архитектуры.

Этот документ опирается на предыдущую полную спецификацию проекта и служит **дорожной картой реализации**.

---

## 2. Общие правила разработки для нейросети

### 2.1 Главное правило

**Нельзя генерировать весь проект одним большим ответом/шагом.**

Нужно идти итеративно:
1. сделать маленькую часть;
2. проверить её;
3. исправить ошибки;
4. только потом переходить дальше.

### 2.2 После каждого шага обязательно

После каждого этапа необходимо:
- проверить TypeScript-ошибки;
- проверить импорт/экспорт файлов;
- проверить маршруты Expo Router;
- проверить, что приложение хотя бы собирается;
- проверить, что нет явных runtime-ошибок;
- проверить консоль/терминал;
- проверить, что новые сущности не ломают старые;
- убедиться, что код соответствует offline-first архитектуре.

### 2.3 Если есть ошибка

Если на этапе появилась ошибка:
- не переходить к следующему этапу;
- сначала локализовать причину;
- исправить ошибку;
- заново прогнать проверку;
- убедиться, что исправление не внесло регрессии.

### 2.4 Что запрещено делать по ходу реализации

Нельзя:
- заменять реальную архитектуру временными mock-обходами;
- складывать бизнес-логику прямо в экран;
- временно хранить бизнес-данные в состоянии компонента вместо БД;
- делать сетевые зависимости «на будущее», которые ломают offline;
- оставлять типы `any`, если можно описать нормальные типы;
- пропускать миграции и seed;
- пропускать самопроверку после этапа.

### 2.5 Формат работы нейросети на каждом шаге

На каждом шаге нейросеть должна действовать так:
1. реализовать только текущий этап;
2. перечислить, какие файлы добавлены/изменены;
3. проверить логическую целостность этапа;
4. мысленно прогнать сборку и найти возможные ошибки;
5. свериться с предыдущей спецификацией;
6. только затем переходить к следующему шагу.

## 3. Глобальная стратегия реализации

Проект нужно собирать в таком порядке:
1. каркас проекта;
2. базовая конфигурация Expo/TypeScript;
3. подключение WatermelonDB;
4. схема БД и модели;
5. seed и локальные данные;
6. repository layer;
7. базовые UI-компоненты;
8. главный сценарий поиска и выбора схем;
9. пошаговый экран схемы;
10. экран отчёта и генерация TXT/PDF;
11. экран истории;
12. админ-раздел;
13. optional sync layer;
14. полировка, тесты, проверки, сборка development build.

Ниже всё разбито на детальные микрошаги.

# ЭТАПЫ РАЗРАБОТКИ

## Этап 0. Подготовка контекста перед генерацией кода

### Цель
Сначала нейросеть должна зафиксировать архитектурные ограничения и не забывать их по ходу реализации.

### Что нужно сделать
- перечитать основную спецификацию;
- зафиксировать, что проект **offline-first / local-first**;
- зафиксировать, что **WatermelonDB — единственное persistent-хранилище бизнес-данных**;
- зафиксировать, что используется **Expo Development Build**, а не Expo Go;
- зафиксировать, что весь код — **TypeScript only**;
- зафиксировать целевую структуру каталогов;
- зафиксировать, что UI-тексты на русском, а типы и код на английском.

### Результат этапа
Нейросеть понимает рамки проекта и дальше не уходит в неправильную архитектуру.

### Самопроверка
Перед переходом дальше проверить:
- не планируется ли использование AsyncStorage для бизнес-данных;
- не планируется ли зависимость от API;
- не забыт ли Expo Router;
- не забыт ли seed;
- не забыт ли admin-раздел;
- не забыт ли экспорт отчётов.

Сделать интерфейс действительно пригодным для работы в полевых условиях.

### Что нужно сделать
Проверить и доработать:
- размеры контролов;
- контрастность;
- фиксированные нижние кнопки действий;
- визуальные статусы;
- читабельность текста;
- режим увеличенных контролов.

### Минимальный результат
Интерфейс явно ориентирован на склад/производство, а не на обычный потребительский мобильный UI.

### Самопроверка
Проверить:
- одной рукой можно выполнить основные сценарии;
- нет мелких touch targets;
- важные статусы очевидны;
- формы лаконичны;
- нет лишней декоративности, мешающей работе.

# 4. Мини-чеклист после каждого этапа

После каждого завершённого этапа нейросеть должна пробегать этот чеклист:

1. Всё ли, что обещано на этапе, реально создано?
2. Все ли новые файлы импортируются корректно?
3. Нет ли ошибок типов?
4. Нет ли битых путей и алиасов?
5. Нет ли очевидных runtime-ошибок?
6. Не вынесена ли важная бизнес-логика в экран случайно?
7. Не нарушен ли offline-first принцип?
8. Не появился ли временный mock вместо настоящей реализации?
9. Не сломаны ли предыдущие этапы?
10. Можно ли переходить дальше без технического долга, который блокирует следующие шаги?

Если хотя бы на один из этих вопросов ответ «нет» — сначала исправить проблему.

# 5. Рекомендуемый режим работы нейросети по этому плану

## Вариант правильного цикла

Для каждого этапа:
1. реализовать только этот этап;
2. показать список созданных/изменённых файлов;
3. кратко пояснить, что именно сделано;
4. выполнить самопроверку;
5. проверить возможные ошибки терминала/сборки;
6. только потом переходить к следующему этапу.

## Вариант неправильного цикла

Неправильно:
- сразу генерировать весь проект;
- пропускать проверки;
- писать «остальное потом» в базовой архитектуре;
- делать вид, что WatermelonDB уже работает, не подготовив schema/models/migrations;
- делать экраны раньше repository/service слоя, если экран зависит от него;
- писать экспорт PDF/TXT до того, как стабилен reportBuilder.

# 6. Итоговый принцип

Разрабатывать этот проект нужно как **цепочку маленьких законченных поставок**, где каждая следующая часть опирается на предыдущую.

Ключевой принцип:

**Сначала стабильный фундамент, потом сценарии, потом экспорт, потом админка, потом полировка.**

Не переходить к следующему шагу, пока:
- текущий шаг не реализован;
- не проверены ошибки;
- не проверена связность;
- не подтверждено, что offline-first архитектура не нарушена.

Этот документ нужно использовать как пошаговую карту реализации для нейросети, чтобы она последовательно довела проект до полноценного рабочего состояния.
