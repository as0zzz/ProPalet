# Спецификация проекта: offline-first мобильное приложение для грузчиков

## 1. Назначение документа

Этот документ описывает **полную техническую спецификацию** мобильного приложения для грузчиков / операторов погрузки.

Документ предназначен для:
- генерации кода нейросетью;
- работы senior/lead разработчика как source of truth;
- проектирования offline-first/local-first архитектуры;
- реализации **полноценного рабочего проекта**, а не демонстрационного макета.

Главная цель: разработать мобильное приложение для управления погрузкой, которое **полностью функционально без интернета** и остается работоспособным даже при отсутствии сети в течение недель.

---

## 2. Контекст и ключевая цель продукта

Приложение используется на производстве / складе / в зоне погрузки. Пользователь — грузчик, оператор погрузки, мастер смены или сотрудник склада.

Сценарий использования:
1. Оператор выбирает модель вагона.
2. Оператор выбирает тип пакета.
3. Приложение по локальной базе находит подходящие схемы погрузки.
4. Оператор проходит схему пошагово, включая многоярусные сценарии.
5. Приложение позволяет сформировать и сохранить отчет.
6. Отчет можно экспортировать в TXT/PDF и поделиться им через системный share sheet.
7. Вся логика, данные, изображения схем и отчеты доступны локально.

Критично: **приложение не должно зависеть от сети ни в UI, ни в бизнес-логике, ни в доступности данных**.

## 3. Принципы архитектуры

### 3.1 Offline-first / local-first

Обязательные принципы:
- приложение должно быть полностью работоспособным без интернета;
- сеть не является обязательной частью ни одного пользовательского сценария;
- все бизнес-данные хранятся локально;
- все справочники, схемы, изображения, отчеты и служебные данные доступны локально;
- UI не должен ожидать сетевых запросов;
- отсутствие сети не должно вызывать падения, вечные loader'ы, блокировки интерфейса или деградацию базовых сценариев;
- синхронизация с сервером допускается только как **опциональный модуль**, который можно отключить без ущерба приложению.

### 3.2 Persistent storage

**WatermelonDB — основное и единственное persistent-хранилище для бизнес-данных.**

Ограничения:
- не использовать AsyncStorage для бизнес-данных;
- не хранить справочники и отчеты в Zustand;
- Zustand разрешен только для легкого UI-state: фильтры, текущий шаг wizard, локальные переключатели режима, ephemeral-state;
- все сущности предметной области, отчеты, настройки, логи, очереди синхронизации — в WatermelonDB.

### 3.3 React Native / Expo

Проект ориентировать на:
- **Expo SDK 55**;
- **React Native**;
- **TypeScript only**;
- **Expo Router** как основную проектную структуру навигации;
- **Expo Development Build**, а не Expo Go.

Причина:
- WatermelonDB и связанные нативные зависимости требуют кастомного native setup;
- Expo Go недостаточен для полного нативного сценария;
- проект должен быть готов к Development Build с учетом необходимых config/plugins/native setup.

### 3.4 Совместимость с новой архитектурой RN

Нужно стремиться к хорошей совместимости с новой архитектурой React Native настолько, насколько это возможно для текущего стека.

Это означает:
- избегать нестабильных или устаревших паттернов;
- не строить архитектуру вокруг legacy-only решений;
- использовать актуальные библиотеки и интеграционные подходы;
- все нативные настройки документировать явно.

## 6. Функциональные требования

### 6.1 Основной сценарий

#### Шаг 1. Выбор модели вагона
Пользователь вводит модель вагона через поиск с автокомплитом по локальной базе.

Требования:
- поиск выполняется локально;
- результаты обновляются реактивно;
- поиск должен быть быстрым;
- строка поиска должна нормально обрабатывать длинные названия;
- после выбора значение фиксируется как selected chip / selected value.

#### Шаг 2. Выбор типа пакета
Пользователь вводит тип пакета аналогичным образом через локальный поиск.

Требования:
- локальный автокомплит;
- debounce;
- нормализация строки поиска;
- реактивные обновления через WatermelonDB observe.

#### Шаг 3. Поиск схем погрузки
После выбора комбинации:
- модель вагона;
- тип пакета;

система мгновенно показывает все подходящие схемы погрузки.

#### Шаг 4. Прохождение схемы
Если схема одноярусная — она показывается сразу.
Если схема многоярусная — показывается wizard по шагам.

#### Шаг 5. Завершение и отчет
После завершения схемы пользователь переходит на экран отчета и может:
- сохранить отчет в БД;
- экспортировать TXT;
- экспортировать PDF;
- поделиться файлом.

## 8. Правила моделирования данных

### 8.1 Обязательные требования
- все таблицы должны иметь понятные индексы там, где это нужно для поиска и связей;
- все сущности должны иметь `createdAt` и `updatedAt`;
- для будущей синхронизации предусмотреть `version`, `syncStatus`, `isDeleted` там, где это уместно;
- любые локальные файлы должны храниться как пути в БД;
- изображения схем — локальные asset references или локальные file URI.

### 8.2 Idempotent seed
Seed должен быть идемпотентным:
- повторный запуск не должен дублировать данные;
- обновление seed-версии должно быть контролируемым;
- инициализация должна быть безопасна при повторном старте приложения.

## 9. Экраны приложения

### 9.1 Главный экран

Назначение: быстрый вход в рабочий сценарий.

Содержимое:
- поиск модели вагона;
- поиск типа пакета;
- автокомплит с debounce;
- реактивное обновление результатов через WatermelonDB observe;
- после выбора — список найденных схем.

Карточка схемы должна включать:
- миниатюру;
- название;
- количество ярусов;
- краткое описание;
- кнопку «Открыть».

Состояния:
- нет введенного запроса;
- ничего не найдено;
- выбран вагон, но не выбран тип пакета;
- комбинация выбрана, но схем нет;
- схемы найдены.

### 9.2 Экран схемы / пошаговой укладки

Сценарии:
- одноярусная схема — показать сразу;
- многоярусная схема — wizard.

Для каждого яруса показать:
- изображение;
- номер яруса;
- прогресс;
- список требуемого инвентаря;
- заметки / инструкции;
- чекбокс «ярус завершён».

Кнопки:
- «Назад»;
- «Далее»;
- «Завершить схему».

Поведение:
- переход вперед запрещен, если обязательное действие по текущему шагу не завершено, если это предусмотрено логикой;
- текущий шаг визуально выделяется;
- завершенные шаги отмечаются явно;
- после последнего шага — переход к экрану отчета.

### 9.3 Экран отчета

Нужно показать отчет в читабельном виде.

Пользователь может:
- сохранить в БД;
- экспортировать в `.txt`;
- экспортировать в `.pdf`;
- поделиться через системный share sheet.

Формат отчета:
- дата и время;
- модель вагона;
- тип пакета;
- выбранная схема;
- список ярусов;
- список использованного инвентаря с количеством и конфигурацией;
- комментарий оператора;
- подпись / ФИО оператора — optional.

### 9.4 Экран истории отчетов

Нужен отдельный экран со списком всех сохраненных отчетов.

Функции:
- список отчетов;
- фильтрация по дате;
- повторное открытие;
- повторный экспорт;
- удаление / архивирование.

### 9.5 Админ-раздел

Раздел может быть скрытым или отдельным.

Функции:
- CRUD для моделей вагонов;
- CRUD для типов пакетов;
- CRUD для схем погрузки;
- CRUD для инвентаря;
- импорт демо-данных из JSON;
- экспорт справочников в JSON;
- сброс БД;
- переинициализация seed-данных;
- просмотр audit log.

Требования:
- формы на React Hook Form + Zod;
- понятные ошибки на русском;
- использовать репозитории и сервисы, а не прямую логику в экранах.

## 10. Поведение в offline-first режиме

### 10.1 Первый запуск
На первом запуске:
1. База данных создается локально.
2. Если база пустая — автоматически выполняется seed.
3. Seed-данные берутся из локальных JSON-файлов или TS-констант.
4. Изображения схем находятся локально в `assets`.

### 10.2 Работа без интернета
Обязательное поведение:
- никаких падений;
- никаких бесконечных loader'ов;
- никаких обязательных API-вызовов;
- любое действие доступно offline;
- все данные доступны локально;
- экспорт отчетов работает локально.

### 10.3 Будущая синхронизация
Нужно заложить архитектурный слой `sync/` как опциональный модуль.

Требования:
- приложение полностью завершено и без сервера;
- синхронизация легко отключается;
- данные готовы к будущему push/pull;
- использовать `version`, `updatedAt`, `isDeleted`, `syncStatus`;
- `sync_queue` реализовать как заготовку;
- не строить UX вокруг предположения, что сервер обязательно будет доступен.

## 12. Архитектурные слои

Код необходимо разделить на слои.

### 12.1 UI layer
Сюда относятся:
- экраны из `app/`;
- UI-компоненты из `src/components/`;
- локальные UI-хуки.

UI layer не должен:
- содержать сложную бизнес-логику;
- напрямую манипулировать низкоуровневыми DB API, если есть repository/service abstraction;
- дублировать логику экспорта, seed, генерации отчета.

### 12.2 Domain / business logic
Сюда относятся:
- логика выбора схем;
- пошаговая логика по ярусам;
- сборка отчета;
- валидация бизнес-сценариев;
- преобразование seed-данных;
- логика синхронизационной готовности.

### 12.3 Repository layer
Repository layer отвечает за:
- query к WatermelonDB;
- observe-потоки;
- выборки, фильтрацию, поиск;
- операции создания/обновления/удаления сущностей;
- инкапсуляцию DB-запросов.

### 12.4 Services layer
Services отвечают за:
- генерацию TXT/PDF;
- шаринг;
- seed orchestration;
- будущую sync orchestration;
- импорт/экспорт справочников.

## 13. Требования к поиску и реактивности

### 13.1 Локальный поиск
Поиск модели вагона и типа пакета реализуется на клиенте.

Обязательные требования:
- использовать WatermelonDB query + observe;
- использовать нормализацию строки поиска;
- учитывать `trim`, `lowerCase`, удаление лишних пробелов;
- корректно обрабатывать длинные названия;
- быстро показывать пустое состояние;
- после выбора поле должно визуально фиксироваться как selected value / selected chip.

### 13.2 Debounce
Автокомплит должен использовать debounce, чтобы не создавать лишние ререндеры и лишние запросы.

### 13.3 Производительность
Поиск должен быть быстрым даже при росте объема справочников.

Рекомендуемые принципы:
- предсказуемые индексы;
- ограничение полей для фильтрации;
- минимизация лишних преобразований в рендере;
- memo/useMemo/useCallback применять осмысленно, а не формально.

## 14. Формы и валидация

Для админских экранов использовать:
- React Hook Form;
- Zod.

Требования:
- все ошибки на русском языке;
- схема валидации отделена от UI;
- валидация переиспользуема;
- формы не должны содержать неструктурированную логику.

Примеры валидации:
- обязательное название модели вагона;
- уникальность имени модели вагона;
- корректность числовых полей;
- корректность римского варианта конструкции I–V;
- обязательность связей между схемой, вагоном и типом пакета.

## 15. Отчеты и экспорт

### 15.1 Требования к отчету

### 15.2 Формат TXT/PDF

Нельзя:

## 16. Seed-данные

Нужно подготовить демонстрационные seed-данные минимум следующего объема:
- не менее 5 моделей вагонов;
- не менее 8 типов пакетов;
- не менее 10 схем погрузки;
- не менее 12 позиций инвентаря;
- связи схема ↔ инвентарь;
- несколько многоярусных схем.

### 16.1 Источники seed
Допускаются:
- локальные JSON-файлы;
- TS-константы;
- комбинированный подход.

### 16.2 Требования к seed
- seed идемпотентный;
- повторный запуск не дублирует записи;
- seed логика отделена от UI;
- можно переинициализировать seed из админ-раздела;
- есть механизм контроля версии seed.

### 16.3 Изображения seed-схем
Изображения схем должны храниться локально в `assets/schemes/`.

## 18. Требования к коду

### 18.1 Общие правила
- писать только на TypeScript;
- включить строгий `tsconfig`;
- без JavaScript-файлов, кроме обязательных конфигурационных файлов, где экосистема этого требует;
- использовать нормальные типы для маршрутов, моделей, DTO, UI props;
- код должен быть пригоден к запуску, а не демонстрационным.

### 18.2 Чистота архитектуры
- не писать всю логику прямо в экранах;
- использовать кастомные hooks;
- использовать репозитории;
- выделять сервисы;
- выделять чистые переиспользуемые компоненты.

### 18.3 Производительность
- минимизировать лишние ререндеры;
- применять `memo`, `useCallback`, `useMemo` осмысленно;
- списки оптимизировать;
- использовать FlashList, где есть длинные списки;
- изображения использовать локально, без сетевых загрузок.

### 18.4 Query policy
Все запросы к WatermelonDB должны идти через repository layer и реактивные observe-потоки там, где это требуется по UX.
